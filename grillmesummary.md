# Grill Session Summary — Full-Hand PokerBot Upgrade

This document captures the complete design agreement from the grill session about upgrading the PokerBot from a single-street snapshot advisor into a full-hand, action-timeline-based heads-up poker assistant.

---

## 1. Core Goal

The current bot analyzes only a **snapshot**:

- hero hole cards
- current board
- pot
- bet to call
- stacks
- position
- villain range

The desired upgrade is for the bot to understand **how the entire hand was played**, from preflop through the current decision point.

The new bot should model:

- starting stacks
- positions
- blinds
- every betting action
- street transitions
- board cards by street
- pot growth
- stack changes
- current player to act
- last aggressor
- villain line/history
- range evolution
- current strategic decision

The core product direction is:

```text
Full Hand Timeline
→ Strict State Engine
→ Built-in Preflop Baseline
→ Range LLM JSON
→ Range Validation/Fallback
→ Equity + Math
→ Decision LLM
→ Structured Results
```

---

## 2. High-Level Architecture Decision

### Agreed direction

The bot should model the hand as a **sequential action history**, not as a disconnected snapshot.

Example target hand representation:

```text
Blinds: 0.5bb / 1bb
Hero BTN/SB AhKd stack 100bb
Villain BB stack 100bb

Preflop:
Hero raises
Villain calls

Flop: Ks 9h 4c
Villain checks
Hero bets
Villain calls

Turn: 2h
Villain checks
Hero bets
Villain raises
Hero ?
```

The timeline should be the source of truth. Pot, stacks, bet to call, effective stack, SPR, and street state should be derived from it.

---

## 3. Input Method

### Decision

Use a **manual street-by-street action builder** for v1.

### Rejected for v1

- Raw poker hand-history parsing from PokerStars/GG/Ignition/etc.
- Fully automatic hand import.

### Reason

Manual structured entry is easier to validate, easier to test, and creates a clean internal data model. Raw hand-history parsing can be added later by converting parsed text into the same internal action timeline.

---

## 4. Heads-Up First, Multiplayer Later

### Decision

v1 should support **heads-up only**.

### Future intent

The design should remain expandable to multiplayer later.

### Important implementation choice

Even though v1 is heads-up, the internal model should still be generic and player/action based:

```json
{
  "players": [
    {"id": "hero", "position": "BTN", "stack_start_bb": 100},
    {"id": "villain", "position": "BB", "stack_start_bb": 100}
  ],
  "actions": [
    {"street": "preflop", "actor": "hero", "type": "raise", "amount_added": 2.0},
    {"street": "preflop", "actor": "villain", "type": "call"}
  ]
}
```

Do **not** hardcode separate fields like:

```text
hero_preflop_action
villain_flop_action
```

The generic model avoids a rewrite when multiplayer support is added later.

---

## 5. Game Type and Units

### Decision

v1 is for **cash games only**.

### No tournament support in v1

Do not include:

- antes
- ICM
- payout pressure
- tournament-specific stack/payout logic

### Units

Everything should be entered and displayed in **big blinds**.

Examples:

```text
Hero stack: 100bb
Villain stack: 100bb
Pot: 7.5bb
Bet: 5bb
Bet to call: 12bb
```

No chip/dollar conversion is needed for v1.

---

## 6. Blind Posting

### Decision

The engine should automatically post heads-up blinds.

Default heads-up cash blinds:

```text
Small blind = 0.5bb
Big blind = 1bb
```

### Position behavior

If Hero is BTN/SB:

```text
Hero posts 0.5bb
Villain posts 1bb
Hero acts first preflop
```

If Hero is BB:

```text
Villain BTN/SB posts 0.5bb
Hero posts 1bb
Villain acts first preflop
```

Blinds should not be manually entered as normal actions in the frontend action list. They are auto-generated by the backend state engine.

---

## 7. Action Model

### Supported action types for v1

```text
fold
check
call
bet
raise
```

### All-in handling

All-in should not be a separate main action type. Instead, use an optional flag:

```json
{
  "type": "raise",
  "amount_added": 73.5,
  "all_in": true
}
```

This supports:

- bet all-in
- raise all-in
- call all-in

while keeping the action model simple.

---

## 8. Action Amount Semantics

### Decision

The internal source of truth is **incremental amount added to the pot**, not “raise to total.”

Example:

Blinds 0.5/1.

Villain is BB and has already posted 1bb. Hero raises to 2.5bb. If Villain calls, Villain adds:

```text
1.5bb
```

If Villain 3-bets to 10bb total, and has already posted 1bb, Villain adds:

```text
9bb
```

### Internal representation

```json
{
  "street": "preflop",
  "actor": "villain",
  "type": "raise",
  "amount_added": 9.0
}
```

### UI helper

Even though the internal model uses incremental chips, the UI should support a **raise-to helper**.

Example:

User enters:

```text
Raise to 10bb
```

UI/backend converts that to:

```text
amount_added = 9bb
```

Optional audit/debug fields can be included:

```json
{
  "type": "raise",
  "amount_added": 9.0,
  "input_mode": "raise_to",
  "input_amount": 10.0
}
```

Backend should treat `amount_added` as the source of truth. If `input_mode/input_amount` are present, backend may validate consistency.

---

## 9. Derived Pot, Stacks, and Betting State

### Decision

The user should no longer manually enter:

- pot
- bet to call
- current stack
- SPR

These must be derived from:

- starting stacks
- automatic blinds
- every action in the timeline

The state engine should compute:

- pot size
- each player’s remaining stack
- each player’s street contribution
- current bet amount
- amount facing the acting player
- bet to call
- legal actions
- minimum raise amount
- effective stack
- SPR
- last aggressor
- current street
- current actor
- whether betting round is closed
- whether hand has ended
- whether hand is all-in locked

---

## 10. Strict Action Validation

### Decision

Use strict validation in both frontend and backend.

### Frontend

The UI should only show/enable legal actions.

### Backend

The backend must replay and validate the full timeline regardless of frontend gating.

This protects against:

- direct API calls
- frontend bugs
- invalid future UI changes
- inconsistent test states

### Examples of invalid sequences

```text
Hero checks
Villain calls
```

Invalid because no bet exists.

```text
Hero bets 10
Villain checks
```

Invalid because Villain is facing a bet.

```text
Hero raises
Villain raises too small
```

Invalid if the raise does not meet minimum raise rules.

### Required validation knowledge

The engine must know:

- whose turn it is
- whether the player is facing a bet
- call amount
- min legal raise
- max amount based on stack
- whether action closes the street
- whether the hand has ended

---

## 11. Street Advancement

### Decision

Betting state should auto-advance.

Flow:

```text
Preflop betting closed
→ await flop cards

Flop cards entered
→ flop actions enabled

Flop betting closed
→ await turn card

Turn card entered
→ turn actions enabled

Turn betting closed
→ await river card

River card entered
→ river actions enabled

River betting closed
→ hand complete
```

The app should not auto-analyze. It should only auto-advance the internal street state when betting is closed.

---

## 12. Manual Analyze Button

### Decision

There should be a manual **Analyze** button. No auto-analysis.

### Button gating

The Analyze button should be strictly gated in the UI and validated in the backend.

Analyze should only be available when:

- Hero has exactly 2 hole cards
- board state is valid for the current street
- it is Hero’s turn to act
- the hand has not ended by fold
- the hand is not all-in locked
- there is a real legal decision available

Examples of disabled reasons:

```text
Waiting for villain action
Enter flop cards
Hand already ended
No decision available: all players are all-in
Invalid board state
```

---

## 13. All-In / No Further Decisions

### Decision

If the hand becomes all-in locked before the river, v1 should **stop decision analysis**.

Do not provide fold/call/raise advice when no action remains.

### Rejected for v1

Equity-only all-in reporting.

That can be added later, but v1 is focused on decision assistance.

---

## 14. Showdown / Known Villain Cards

### Decision

v1 should be **decision-only**.

Do not enter or use known villain hole cards.

### Future feature

A later review/training mode could allow entering villain’s actual hand after showdown to compare:

- Was villain’s hand inside the predicted range?
- Was Hero’s decision good?
- Did the range analysis miss something?

But this is out of scope for v1.

---

## 15. Villain Profile

### Decision

Use a simple villain profile dropdown for v1.

No free-text notes yet.

### Profile list

Recommended profiles:

```text
Unknown
Nit
Tight Passive
Tight Aggressive
Loose Passive / Calling Station
Loose Aggressive
Maniac
Solid Reg
Weak Recreational
```

`Unknown` should be the default.

### Reason

A dropdown gives the Range LLM useful context while keeping the prompt controlled, structured, and fast.

Free-text notes can be added later.

---

## 16. Villain Range Evolution

### Decision

Use an LLM to estimate villain’s current range at the decision point.

### Important constraint

Do **not** call the LLM after every action.

Instead:

1. User builds the full hand timeline locally.
2. Backend derives the current hand state.
3. At the decision point, make one Range LLM call.
4. Then compute math/equity.
5. Then make one Decision LLM call.

This keeps the number of LLM calls fixed at two per analysis.

---

## 17. Two-LLM-Call Pipeline

### Decision

At the decision point, use exactly two LLM calls:

```text
Timeline + derived facts
→ Range LLM
→ estimated range JSON
→ code validates/parses range
→ equity/math
→ Decision LLM
→ final action
```

### Call 1: Range LLM

Purpose:

- interpret villain’s line
- estimate current villain range
- categorize range into value/draws/bluffs
- assign coarse likelihood labels
- output strict JSON

### Call 2: Decision LLM

Purpose:

- use full timeline
- use range analysis JSON
- use computed equity/math
- recommend exactly one action:
  - FOLD
  - CALL
  - RAISE

---

## 18. Rule-Based vs LLM Range Deciding

### Discussion result

A pure rule-based range engine is fast and testable but rigid.

A pure LLM range engine is flexible and good at interpreting narrative betting lines, but can hallucinate invalid ranges and fake precision.

### Final v1 decision

Use:

```text
Built-in preflop baseline ranges
+ LLM post-action range interpretation
+ backend validation/fallback
```

So v1 is not fully rule-based. The LLM decides current ranges, but code provides guardrails.

---

## 19. Built-In Preflop Ranges

### Decision

Use simple heads-up preflop presets for v1.

### Scope

Use broad ranges, not detailed sizing-dependent matrices.

Recommended presets:

```text
BTN/SB open range
BB defend vs BTN/SB open
BB 3-bet range
BTN/SB call vs BB 3-bet
BTN/SB 4-bet range
BB call vs 4-bet
```

### Rejected for v1

Detailed preflop matrix by exact sizing, such as:

```text
BTN open 2bb
BTN open 2.5bb
BTN open 3bb
BB defend vs 2bb
BB defend vs 2.5bb
BB defend vs 3bb
...
```

The Range LLM can adjust qualitatively based on sizing/action details for v1.

---

## 20. Range LLM Output Format

### Decision

Range LLM should return structured JSON.

### Recommended schema

```json
{
  "estimated_range": "TT+, AQs+, KQs, QJs, JTs",
  "confidence": "LOW|MEDIUM|HIGH",
  "overall_tendency": "value-heavy|balanced|draw-heavy|bluff-heavy|uncertain",
  "categories": {
    "value": {
      "range": "99,44,22,K9s",
      "weight": "HIGH|MEDIUM|LOW|NONE",
      "reasoning": "Why these hands are likely"
    },
    "draws": {
      "range": "QhJh,JhTh,AhQh",
      "weight": "HIGH|MEDIUM|LOW|NONE",
      "reasoning": "Why draws are present"
    },
    "bluffs": {
      "range": "AQo,AJo",
      "weight": "HIGH|MEDIUM|LOW|NONE",
      "reasoning": "Why bluffs are present"
    }
  },
  "reasoning": "Short explanation of how villain's line shaped the range",
  "fallback_used": false
}
```

### Main requirement

`estimated_range` must be parseable by the existing/future range parser.

The equity engine uses `estimated_range` as the main range.

Categories are primarily for display and reasoning in v1.

---

## 21. Range Categories and Weights

### Decision

Use coarse category weights, not exact decimal weights from the LLM.

Accepted labels:

```text
HIGH
MEDIUM
LOW
NONE
```

### Reason

Fast LLMs should not be trusted to provide exact combo frequencies like `0.37` or `0.82`. Those numbers may look precise but are usually speculative.

Coarse labels are safer:

```text
HIGH = likely/frequent
MEDIUM = plausible/sometimes
LOW = possible but discounted
NONE = absent/negligible
```

### v1 equity behavior

Do **not** use weights in equity calculation for v1.

Use only:

```text
estimated_range
```

Categories and weights are displayed and passed to the Decision LLM for qualitative reasoning.

Weighted equity can be added later.

---

## 22. Invalid Range LLM Output Handling

### Decision

If the Range LLM returns invalid or unparseable output, do **not** make a third repair LLM call in v1.

Instead:

- validate output
- if invalid, use fallback built-in default/current range
- mark confidence LOW
- set fallback warning
- continue to math and final Decision LLM

### Reason

The goal is to keep two LLM calls under roughly 20 seconds. A third repair call could break that.

### Fallback JSON example

```json
{
  "estimated_range": "<broad fallback range>",
  "confidence": "LOW",
  "overall_tendency": "uncertain",
  "categories": {},
  "reasoning": "Range LLM failed or returned invalid output; using broad preflop-derived fallback.",
  "fallback_used": true
}
```

---

## 23. Range JSON Validation Rules

Backend should validate:

- response is valid JSON
- required fields exist
- `estimated_range` exists
- `estimated_range` parses successfully
- `confidence` is one of:
  - LOW
  - MEDIUM
  - HIGH
- `overall_tendency` is one of:
  - value-heavy
  - balanced
  - draw-heavy
  - bluff-heavy
  - uncertain
- category weights, if present, are one of:
  - HIGH
  - MEDIUM
  - LOW
  - NONE

Invalid category ranges should not necessarily fail the whole analysis if `estimated_range` is valid. The main equity range matters most.

---

## 24. Final Decision LLM Context

### Decision

The final Decision LLM should receive:

- full action timeline
- board cards
- hero hole cards
- positions
- villain profile
- derived hand state
- full Range LLM JSON
- fallback warning if used
- equity and math metrics

It should not receive only the final range string.

### Reason

The final advisor needs context to explain the decision clearly and to mention sensitivity to range assumptions.

---

## 25. Decision LLM and Range Disagreement

### Decision

The final Decision LLM may **critique or sensitivity-check** the Range LLM output, but it must base the main recommendation on the computed range/equity.

Allowed:

```text
Using the estimated range, this is a call. If villain is significantly tighter than estimated, folding becomes better.
```

Not allowed:

```text
I ignore the provided range and assume villain only has sets, so fold.
```

The range analysis and computed math are the source of truth for the main recommendation.

---

## 27. Raise Recommendation Output

### Decision

If the bot recommends RAISE, display both:

- total raise-to size
- incremental amount to add

Example:

```text
RAISE
Raise to: 18bb total
Amount to add: 12bb
```

### Reason

Poker players understand “raise to” naturally, but the internal engine tracks incremental `amount_added`.

---

## 28. LLM Model Strategy and Time Budget

### User concern

Two LLM calls should not exceed around 20 seconds.

### Recommendation

Use different models/settings for the two calls.

### Range LLM

Use a fast/cheap model.

Recommended settings:

```text
timeout: 8–10 seconds
max_tokens: 500–700
temperature: 0.2
output: strict JSON only
```

The Range LLM does not need elegant prose. It needs fast, structured output.

### Decision LLM

Use a stronger reasoning model.

Recommended settings:

```text
timeout: 12–15 seconds
max_tokens: 900–1200
temperature: 0.3
output: structured decision format
```

Total target:

```text
Range LLM: ~8s
Math/equity: fast enough
Decision LLM: ~12s
Total: ~20s target
```

---

## 29. Timeout Behavior

### Range LLM timeout

If Range LLM times out:

- use fallback range
- set confidence LOW
- set `fallback_used = true`
- continue to equity/math
- continue to final Decision LLM

### Decision LLM timeout

If Decision LLM times out:

- return hand state
- return range analysis
- return math metrics
- set decision action to ERROR
- show clear reason

Example:

```json
{
  "action": "ERROR",
  "reasoning": "Final advisor timed out",
  "confidence": "N/A",
  "raise_size": "N/A"
}
```

Do not silently invent a final action in deterministic code.

---

## 30. Equity Behavior

### v1 decision

Use unweighted equity against `estimated_range`.

Ignore category weights in the equity calculation for now.

### Recommended simulation counts

Adaptive simulations:

```text
preflop: 8,000–10,000
flop: 10,000
turn: 12,000
river: exhaustive evaluation
```

If speed becomes an issue, reduce default to around 5,000 simulations and add a future “high accuracy” mode.

## 33. Frontend State vs Backend Session

### Decision

Use frontend-only timeline state until Analyze is clicked.

### Flow

```text
User builds hand in browser
→ browser stores setup/actions/board locally
→ user clicks Analyze
→ frontend sends full hand object to backend
→ backend replays and validates everything statelessly
→ backend returns analysis
```

### Rejected for v1

Backend session state where every action is sent to the server as it happens.

### Reason

Frontend-only state is simpler, stateless, easier to test, and avoids session bugs.

Backend still performs full replay validation on Analyze.

## 38. Decision Prompt Requirements

The Decision LLM prompt should include:

- full hand timeline
- current hand state
- hero cards
- board cards
- villain profile
- range analysis JSON
- fallback warning if applicable
- computed equity
- pot odds
- EV call
- MDF
- SPR
- outs
- legal actions available

It should instruct the LLM:

- recommend exactly one action
- base main recommendation on computed metrics/range
- may include sensitivity note
- may critique range assumptions but cannot replace the source-of-truth range
- if raising, provide both raise-to total and amount-to-add

Recommended output format:

```text
REASONING:
<step-by-step reasoning>

SENSITIVITY_NOTE:
<how decision changes if range assumptions are wrong, or N/A>

DECISION: <FOLD|CALL|RAISE>
RAISE_TO_BB: <number or N/A>
AMOUNT_TO_ADD_BB: <number or N/A>
CONFIDENCE: <LOW|MEDIUM|HIGH>
```

## 41. Final Agreed Product Definition

The upgraded PokerBot should become a **heads-up cash-game full-hand advisor**.

The user manually builds the hand street by street. The frontend strictly guides legal actions. The backend replays and validates the entire hand from auto-posted blinds through the current decision point.

At analysis time, the system performs:

1. State derivation from the full timeline.
2. Built-in preflop baseline selection.
3. Fast Range LLM call producing structured JSON.
4. Range validation and fallback if needed.
5. Equity and poker math calculation using the estimated range.
6. Final Decision LLM call using the full timeline, range analysis, and computed math.
7. Structured UI display of hand state, range analysis, math, and final decision.

The design intentionally avoids fake precision from LLM-generated exact weights, keeps the system fast enough for a roughly 20-second target, and preserves future expandability toward multiplayer support.
